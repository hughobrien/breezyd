## Protocol-level findings (independent of param semantics)

- **Auth failure response:** When the password is wrong, the device replies with function `0x07` and a 2-byte payload `01 <byte>`. When the password is correct, function is `0x06`. The Go client should treat function `0x07` as `ErrAuth` rather than passing it through.
- **Discovery is unauthenticated:** Sending any request with `DEFAULT_DEVICEID` returns the device's true 16-char ID and unit type *regardless of password*. This means discovery and password-recovery via search both work even when you don't know the protocol password.
- **High-byte page mechanism:** The protocol's `0xFF` byte in a payload prefixes a "high byte" for the next param ID, allowing access to params at addresses `0xHH00..0xHHFB`. To read VOC at `0x0320`, send payload `FF 03 20`. Pages confirmed populated on Breezy (unit type 0x0011): page 0x00 (~50 params, the main set), page 0x03 (~9 params including VOC at 0x0320), page 0x04 (~10 params, appear to be config/calibration values). Pages 0x01, 0x02, 0x05 returned no useful content.
- **Time encoding is consistent across the device:** all multi-byte time fields use little-endian "smaller unit first" byte ordering. 2-byte durations are `[minutes, hours]`. 3-byte countdowns and the RTC are `[seconds, minutes, hours]`. The Go client should encode/decode all time fields the same way.
- **High-page writes work with `FF <page> <id> <value>` payload:** Validated 2026-05-03 by writing `0x0315 = 0` (VOC sensor disable) using payload `FF 03 15 00` with function `0x03`. The high-byte page-switch mechanism applies symmetrically to writes — same shape as reads. This unlocks writing to night/turbo durations (`0x0302`/`0x0303`) and other page-3/page-4 params programmatically.
- **Per-unit fan-speed scaling differs:** the manual-percentage-to-`0x4A`-value coefficient is around 53.4 on the playroom unit (5340 at 100% manual) and around 59 on the office unit (4260 at 72% manual). Each device has its own motor/impeller calibration. The Go client should expose `0x4A`/`0x4B` as raw values and let consumers compute "% of max" if needed.
- **Fans need ~10-15s to settle after a write.** Reading 0x4A/0x4B sooner shows in-transition values that look noisy or wrong. The poller should not query immediately after a write.

## Datasheet-derived facts (Breezy 160-E Smart, v 2024-01)

- **4 temperature sensors** placed at outdoor / supply / indoor / exhaust positions. Mapped to `0x1F`-`0x22` respectively.
- **VOC/CO2eq combined sensor** (metal-oxide). True CO2 not reported; the "CO2" reading at `0x27` is eCO2 derived from VOC. Real VOC at `0x0320`.
- **Light sensor** for auto-brightness (drives the auto-mode at `0x0402`). The reading itself probably exists in some unmapped param.
- **Frost protection**: the heater can activate automatically based on temperature, not only from the user's enable flag. Explains the `0x68` (config) vs `0x81` (currently-running) distinction.
- **"Auto mode based on sensors"** is a 4th airflow mode beyond recovery/supply/exhaust. Likely `0xB7 = 0` or `4`; not yet tested.
- **Rated airflow**: 57 m³/h normal, 70 m³/h boost on the Breezy 160-E. Full-speed `0x4A` ≈ 5340 corresponds roughly to the 70 m³/h boost on playroom; per-unit scaling differs.
- **Rated heat recovery efficiency**: up to 88%. Useful as a sanity range for any computed Prom efficiency gauge.
- **G3 filter**, replacement interval defaults to 90 days (`0x63`).
- **Heater power**: 100 W. **Total unit power**: up to 22 W (excluding heater).
