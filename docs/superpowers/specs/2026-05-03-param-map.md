# Twinfresh Breezy parameter map (unit type 17 — Breezy 160)

**Date:** 2026-05-03
**Status:** complete; reconciled against the vendor protocol manual `breezy-manual-vendor.pdf` (in this directory) on 2026-05-03.

## Sources

- **Authoritative:** Vendor protocol manual ("Breezy / Breezy Eco — Connection Instruction" — Ventilation-system.com PDF in this directory). Documents packet structure, function codes, special command bytes, and a full parameter table.
- **Supplementary:** Vendor datasheet (also in this directory) — confirms physical design (4 temp sensors at outdoor / supply / indoor / exhaust positions, VOC/CO2eq combined sensor, light sensor, electric heater with frost-protection logic).
- **Empirical:** Live read/write/diff sessions on three operator-owned units, captured via `tools/probe.py` and `tools/snapshot.py`. Operator drove app-side settings and reported observed effects.

## Protocol-level findings

- **Transport:** UDP/4000, max 256-byte packets.
- **Frame format** (per manual): `FD FD | TYPE(0x02) | SIZE_ID(0x10) | ID(16 ASCII) | SIZE_PWD | PWD | FUNC | DATA | Chksum_L | Chksum_H`. Checksum = sum of bytes from `TYPE` through end of `DATA`.
- **Function codes:** `0x01` read, `0x02` write-no-response, `0x03` write-with-response, `0x04` increment, `0x05` decrement, `0x06` controller response.
- **Auth-failure response (undocumented but reproducible):** wrong password → device replies with function `0x07` and 2-byte payload `01 <byte>`. The Go client should treat function `0x07` as a typed `ErrAuth`.
- **Discovery is unauthenticated:** sending any request with the literal device ID `DEFAULT_DEVICEID` returns the device's true 16-char ID (param `0x7C`) and unit type (`0xB9`) regardless of password.
- **Special command bytes inside DATA** (per manual): `0xFC` change FUNC, `0xFD` "parameter not supported" marker (from device), `0xFE <size>` next param value uses `<size>` bytes, `0xFF <hi>` change parameter-number high byte (page).
- **Parameter pages observed:** page 0 (the main set), page 0x01 (`0x0128`/`0x0129`/`0x012A`), page 0x03 (timer durations + VOC sensor block + schedule status + frost), page 0x04 (display config block). Pages 0x02 and 0x05+ returned only "unsupported" responses on these units.
- **Time encoding:** little-endian "smaller unit first." 2-byte = `[min, hr]`. 3-byte = `[sec, min, hr]`. 4-byte = `[min, hr, day_lo, day_hi]` (filter remaining and motor running hours). 6-byte schedule entries follow the manual's documented layout.
- **Multi-byte writes** use `FE <size> <id> <bytes...>` in the DATA block. We validated this format with a write to `0x0315` (page-3, 1-byte) using payload `FF 03 15 00`. Multi-byte writes to e.g. `0x0302` (night duration) follow the same pattern with explicit size.
- **Fan-speed settle time:** ~10-15 s after a write to `0x02`, `0x44`, or `0xB7`. Reading `0x4A`/`0x4B` sooner returns transitional values. The poller should not re-read the just-written param immediately.
- **Per-unit max fan RPM differs:** at 100% manual the playroom reads `0x4A=5340`, the office reads `0x4A≈5900` (extrapolated from 4260 at 72%). This is just different physical motors — `0x4A`/`0x4B` are real RPM, not "vendor units."
- **Sensor over-threshold drives the fan even in manual mode:** when humidity / CO2 / VOC exceed their thresholds AND the corresponding sensor is enabled, the device boosts the fan to a sensor-driven level regardless of the user's manual %. To run at a lower-than-sensor speed, disable the sensor or raise its threshold. The override is reflected in `0x84` (alert bitmap) and `0x4A`/`0x4B` (live fan RPM).

## Security note

The device exposes its protocol password (`0x7D`), the WiFi SSID (`0x95`), and the WiFi password (`0x96`) **in cleartext** to any LAN client that knows the device ID. The device ID itself is broadcastable via the unauthenticated `DEFAULT_DEVICEID` discovery. Network-segmenting these onto an IoT VLAN is the appropriate mitigation — not in scope for this project.

## Mistakes worth remembering

- The `0x002B` / `0x0024` per-unit-different values that took several investigations to characterize: `0x24` is **RTC battery voltage** (in the manual, mV); `0x2B` is **undocumented**. I kept theorizing about pressure / RPM / RSSI before checking the manual.
- A run of "perfect" matches at `0x6C/6D/6E = 50/51/52` looked like humidity min/avg/max but was actually checksum bytes incrementing 1:1 with parameter ID. False positive. The lesson: parse responses by their structure (`FE`/`FD`/value framing) rather than grepping target bytes across raw packets.
- Originally I labeled `0x9A` as "WiFi channel of the connected AP." It's actually the AP-mode configured channel, irrelevant in client mode. Confirmed by checking three units on different APs all reading 11.
- I gave up on the schedule contents and the recovery-efficiency rating early — both are documented in the manual; I should have looked there first.
